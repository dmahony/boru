# Docs-based small-room messaging latency spike plan

Goal: Measure whether an iroh-docs-backed message log is fast enough for 1:1 and small group rooms in `boru-chat`, before replacing or augmenting the current gossip-only message path.

## Context from the current repo

- Current chat messages are signed `chat_core::Message::Message` payloads sent with `GossipSender::broadcast` from `examples/iced_chat/app.rs` and `examples/chat.rs`.
- Room identity is already a stable `TopicId`; room history is persisted in `src/room_history.rs` and opened from the iced chat-list flow in `examples/iced_chat/app.rs`.
- `iroh-docs` is available on crates.io as `iroh-docs = 0.101.0` and should be added only in the spike branch/prototype, not to the production path until measurements justify it.

## Spike questions

1. What is the end-to-end delay from `send pressed` on one peer to `message visible` on every other peer when messages are appended to an iroh doc?
2. How does latency change for 1:1, 3-peer, 5-peer, and 10-peer rooms?
3. How does latency change under local LAN/direct conditions, relay-only/NAT-like conditions, and artificially delayed/lossy links?
4. How much overhead comes from doc setup/subscription versus steady-state message append/retrieval?
5. Does the docs path preserve a usable room-open UX, or does initial sync make room entry feel slow?

## Metrics to capture

Capture every sample as structured JSONL so later workers can analyze percentiles without re-running the spike.

Required fields per sample:

- `run_id`: unique benchmark run id.
- `scenario`: e.g. `local_1v1`, `local_10_all_senders`, `netem_100ms_1pct_5_peers`.
- `room_id`: current `TopicId` or derived doc namespace id.
- `sender_peer`: peer public key/short id.
- `receiver_peer`: peer public key/short id.
- `message_id`: monotonic per-room id generated by the sending peer.
- `payload_bytes`: encoded payload size.
- `send_started_at_ms`: before local doc append call.
- `append_done_at_ms`: after local doc append returns.
- `remote_observed_at_ms`: when receiver observes the doc update.
- `visible_at_ms`: when receiver has decoded the entry and would push it into the UI log.
- `peer_count`: total room participants.
- `concurrent_senders`: number of peers sending during the round.
- `transport_mode`: `direct`, `relay`, or `simulated_delay`.
- `success`: boolean.
- `error`: null or a short error string.

Derived metrics:

- Local append latency: `append_done_at_ms - send_started_at_ms`.
- End-to-end observable latency: `remote_observed_at_ms - send_started_at_ms`.
- End-to-end visible latency: `visible_at_ms - send_started_at_ms`.
- Fanout completion latency: max visible latency across receivers for one message.
- Delivery ratio: received messages / expected receiver-message pairs.
- Duplicate ratio: duplicate observations / unique messages.
- Startup sync time: room open start to first complete historical message set.

Report at least p50, p90, p95, p99, max, and delivery ratio for each scenario.

## Prototype surface area

Build the smallest headless prototype first; avoid touching production UI until the numbers look promising.

The prototype must expose these operations:

1. `create_doc_for_room(topic: TopicId) -> DocHandle`
   - Creates or opens the iroh doc namespace associated with a room topic.
   - The mapping can be deterministic for the spike, but document exactly how peers discover the same doc.

2. `append_message(doc, signed_message, metadata) -> MessageId`
   - Appends the same signed chat payload currently produced by `SignedMessage::sign_and_encode`.
   - Stores sender id, logical message id, wall-clock timestamp, and payload bytes.
   - Returns after the local append is durably/locally visible.

3. `subscribe_messages(doc) -> Stream<ObservedMessage>`
   - Emits historical entries needed to populate the room log.
   - Emits new entries when remote peers append.
   - Deduplicates by `(sender_peer, message_id)`.

4. `retrieve_messages(doc, since) -> Vec<ObservedMessage>`
   - Allows room-open catch-up measurement independent of the live stream.
   - Supports loading the most recent N messages for the room.

5. `bench_docs_room --peers N --senders M --rounds R --payload-bytes B --scenario NAME --out results.jsonl`
   - Runs the measurement without iced or ratatui.
   - Emits the JSONL schema above.

## Minimal integration hook into room open flow

After the headless prototype works, add one narrow optional hook in the iced room-open path:

- Location: `examples/iced_chat/app.rs`, inside `AppMessage::CreateNewRoom`, `AppMessage::OpenRoom(topic)`, and `AppMessage::JoinFromTicket` after the gossip subscription succeeds and before `RoomOpened` is emitted.
- Hook shape: create a `DocsRoomHandle` for the selected `TopicId`, start a docs-message subscription task, and send docs observations into the existing `net_tx`/`AppMessage::NetEvent` path only when the `--docs-chat-spike` flag or feature is enabled.
- Do not remove existing `gossip.subscribe(...)` or `GossipSender::broadcast(...)` behavior during the spike.
- Keep room history (`src/room_history.rs`) unchanged; it should still key rooms by `TopicId`.
- If docs setup fails, log and continue with current gossip behavior so the spike cannot break normal room open/join.

This hook is only for measuring room-open sync and UI-visible latency. The first measurement implementation should remain headless.

## Test scenarios

### Baseline local scenarios

1. `local_1v1_single_sender`
   - 2 peers on one machine/process group.
   - 1 sender, 1 receiver, 100 messages, 128-byte payloads.

2. `local_3_single_sender`
   - 3 peers, 1 sender, 2 receivers, 100 messages.

3. `local_5_single_sender`
   - 5 peers, 1 sender, 4 receivers, 100 messages.

4. `local_10_single_sender`
   - 10 peers, 1 sender, 9 receivers, 100 messages.

5. `local_10_all_senders`
   - 10 peers, all peers send one message per round for 50 rounds.
   - Measures concurrent writer behavior and fanout completion.

### Payload-size scenarios

Run the 1:1 and 10-peer single-sender cases with:

- 128 bytes: normal short chat message.
- 1 KiB: long chat message.
- 16 KiB: stress case, not a target for typical chat but useful for slope.

### Network-condition scenarios

Use either Linux `tc netem` around separate peer processes or a simulator/harness if direct netem is too intrusive.

1. `netem_50ms_0loss_5_peers`
   - 50 ms one-way delay, no loss.

2. `netem_100ms_0loss_5_peers`
   - 100 ms one-way delay, no loss.

3. `netem_100ms_1pct_loss_5_peers`
   - 100 ms one-way delay, 1% packet loss.

4. `relay_like_5_peers`
   - Force relay-only/NAT-like routing if iroh exposes an easy test knob; otherwise mark as skipped with reason.

### Startup / room-open scenarios

1. `open_empty_room`
   - Measure docs setup and subscription time with no messages.

2. `open_room_100_history`
   - Preload 100 messages, measure time to retrieve/decode/render-ready all messages.

3. `open_room_1000_history`
   - Preload 1000 messages, measure catch-up behavior and memory use.

## Success criteria

Treat these as provisional thresholds for a small-room chat UX:

- Local/direct 1:1, 128-byte messages: p95 visible latency <= 250 ms; p99 <= 500 ms.
- Local/direct 5-peer room: p95 fanout completion <= 750 ms; delivery ratio >= 99.9%.
- Local/direct 10-peer room with one sender: p95 fanout completion <= 1.5 s; delivery ratio >= 99.5%.
- Local/direct 10-peer all-senders: p95 visible latency <= 2.5 s; delivery ratio >= 99.0%; no unbounded queue or memory growth.
- 100 ms simulated delay, 5 peers: p95 fanout completion <= 3.0 s; delivery ratio >= 99.0%.
- Open room with 100 historical messages: render-ready within 1.0 s after room selection.
- Open room with 1000 historical messages: render-ready within 3.0 s, or prototype must support lazy loading before production consideration.
- No duplicate visible messages after deduplication.
- Existing gossip chat open/join/send flows still work with the spike feature disabled.

## Measurement execution plan

1. Add `iroh-docs = "0.101.0"` behind a spike-only feature or in a standalone bench/example target.
2. Build a headless `examples/docs_room_latency.rs` or `src/bin/docs_room_latency.rs` harness with in-process peer orchestration if possible.
3. Reuse existing `TopicId`, `SignedMessage`, and chat message payloads so the spike measures the likely production encoding.
4. Emit JSONL samples and one summary table per run.
5. Run the baseline local matrix first; do not start UI integration until those runs produce stable results.
6. Add simulated network conditions after local results are trustworthy.
7. Add the optional iced room-open hook only to measure startup sync/visible latency in the real UI flow.
8. Write a follow-up report with raw result file paths, percentile tables, and a recommendation: adopt docs path, keep gossip path, or use hybrid gossip-for-live plus docs-for-history.

## Risks and watchpoints

- Clock skew can corrupt cross-process latency numbers. Prefer one process with monotonic `Instant` offsets for local runs, or include clock-sync calibration if using multiple processes.
- The doc discovery story may dominate room-open latency. Keep doc creation/discovery timing separate from steady-state message append timing.
- Concurrent writers can create ordering ambiguity. Measure delivery and deduplication first; defer total ordering decisions unless latency is acceptable.
- Avoid making `iroh-docs` a normal dependency until the spike proves value; the current crate is focused on gossip and examples.
- Current room switching is still gossip-oriented; docs integration should be behind an explicit spike flag/feature and should not change normal room lifecycle.

## Deliverables from the measurement phase

- Raw JSONL results for every scenario.
- A summary markdown report with percentile tables and delivery ratios.
- A short recommendation with one of three outcomes:
  - `adopt`: docs latency meets thresholds and integration complexity is acceptable.
  - `hybrid`: use gossip for live messages and docs only for history/catch-up.
  - `reject`: docs latency or complexity is not suitable for small-room chat.
