CARD 1: DISCOVERY SECRET & NAMESPACE DERIVATION
================================================================================

Title: Discovery secret generation and namespace derivation
Status: TODO
Priority: HIGH
Dependencies: Card 0 (Repository Audit)

Objective:
Define the discovery secret type, its generation, and the straightforward
namespace derivation for private-room DHT lookups.

Background:
The public-room discovery system already uses a deterministic discovery
key derived from room name + network + version (see public_room.rs).
Private rooms need a RANDOM discovery key (32 bytes) so that only peers
who have been given the key (via ticket) can find the room on the DHT.
The namespace for DHT operations is simply NamespaceId::new(secret).

Tasks:
1. Define a newtype DiscoverySecret([u8; 32]) in src/discovery_backend.rs
   or a new module src/discovery_secret.rs with:
   - DiscoverySecret::generate() -> Self using getrandom
   - DiscoverySecret::from_bytes([u8; 32]) -> Self
   - DiscoverySecret::as_bytes() -> &[u8; 32]
   - Display/hex Debug (partial, show first 4 bytes only)
   - Into<NamespaceId> or as_namespace_id() convenience method

2. Add a domain-separated namespace derivation:
   Since the discovery secret IS the namespace key (unlike public rooms
   which use domain-separated hashing), the derivation is trivial:
   namespace = NamespaceId::new(discovery_secret.as_bytes())
   No hashing is needed because the secret is already a uniform random
   32-byte value.  Document this clearly.

3. Write tests:
   - generate() produces non-zero output
   - generate() is non-deterministic (two calls produce different values)
   - from_bytes round-trips
   - as_namespace_id() produces the expected value
   - Send + Sync for the new type
   - Deterministic from_bytes returns same secret for same input

Deliverables:
- New type: DiscoverySecret in src/discovery_secret.rs (or added to
  discovery_backend.rs)
- Module registered in src/lib.rs
- Unit tests for generation and round-trip

Acceptance criteria:
- DiscoverySecret::generate() creates a cryptographically random 32-byte
  value.
- as_namespace_id() returns NamespaceId::new(secret_bytes).
- All unit tests pass.
- No changes to existing code (this is additive).

CARD 4: PRIVATE ROOM IDENTITY — ADAPT PUBLIC FOR PRIVATE USE
================================================================================

Title: Private-room identity struct for DHT tracker integration
Status: TODO
Priority: HIGH
Dependencies: Card 1 (Discovery Secret), Card 3 (Room Store Migration)

Objective:
Create a PrivateRoomIdentity struct (analogous to PublicRoomIdentity) that
derives a DHT discovery namespace from the discovery secret and pairs it
with the gossip TopicId.

Architecture note:
For private rooms, the discovery key for DHT operations is simply the
discovery secret itself (no hashing needed — the secret is random).
The TopicId remains whatever the room creator chose (could be a stable
topic derived from the secret, or purely random).

FINAL DESIGN DECISION (Card 4 confirmation):
For the initial private-room DHT implementation, keep the gossip TopicId as
its existing independent random value and add a separate independent random
DiscoverySecret.  The ticket carries both values:
  - topic: TopicId (as today)
  - discovery_secret: Option<DiscoverySecret> (new)
  - peers: Vec<EndpointAddr> (as today)

The DiscoverySecret is used directly as the DHT namespace key.  It is not
hashed with, derived from, or otherwise coupled to TopicId.  Do not use a
single room secret or domain-separated derivation for this implementation.
That alternative is explicitly deferred to future work if a later protocol
version needs it.

This decision preserves the current random TopicId and existing gossip
semantics, keeps the two namespaces independent, and minimizes migration
and compatibility risk.  New private-room modules must repeat this rationale
in their module-level documentation comments; normal logs and Debug output
must not reveal the DiscoverySecret.

Tasks:
1. No new identity struct needed.  The private-room tracker uses:
   - The existing TopicId for the gossip mesh
   - DiscoverySecret for the DHT namespace
   - Both are independent random values

2. Document the rationale in a module doc comment.

Deliverables:
- Design decision documented
- No code change (card may be empty aside from confirmation)

Acceptance criteria:
- Design agreed: separate random TopicId and DiscoverySecret.

CARD 5: PRIVATE-ROOM TRACKER WRAPPER
================================================================================

Title: PrivateRoomTracker — publish/discover wrapper for private DHT rooms
Status: TODO
Priority: HIGH
Dependencies: Card 1 (Discovery Secret), Card 4 (Private Room Identity)

Objective:
Create a thin wrapper that adapts the existing PublicRoomTracker for
private-room use.  Since PublicRoomTracker::new() already accepts an
explicit PublicRoomIdentity, and MainlineDhtBackend operates on
NamespaceId, the private-room tracker only needs a convenience constructor
that bridges DiscoverySecret -> NamespaceId.

Tasks:
1. Create src/private_room_tracker.rs containing:
   - PrivateRoomTracker struct (wraps PublicRoomTracker OR directly wraps
     TopicDiscoveryBackend like PublicRoomTracker does).

   Option A (wrapper): PrivateRoomTracker holds a PublicRoomTracker
     internally.  Its publish_once() and discover_once() delegate down.
     This is the cleanest — no duplication.

   Option B (standalone): PrivateRoomTracker duplicates the
     PublicRoomTracker logic but uses DiscoverySecret as identity.
     This is more explicit but duplicates code.

   RECOMMENDED: Option A — PrivateRoomTracker wraps PublicRoomTracker
   with a PrivateRoomIdentity that is constructed from TopicId +
   DiscoverySecret.

   Wait — PublicRoomTracker expects PublicRoomIdentity which has topic
   and discovery_key fields.  If we construct a PublicRoomIdentity from
   the private room's topic and discovery secret, the tracker would work
   unchanged.  But the discovery_key in PublicRoomIdentity is a [u8; 32],
   same as DiscoverySecret.  So:

   let identity = PublicRoomIdentity::new(topic, discovery_secret.as_bytes());
   let tracker = PublicRoomTracker::new(backend, identity, endpoint_id, secret_key);

   This works without any new code!  PublicRoomIdentity is just topic +
   [u8; 32] discovery_key.

   Create PrivateRoomTracker as a newtype around PublicRoomTracker with
   one extra method: discovery_secret() -> &DiscoverySecret, and an
   ownership-preserving shutdown.

2. Add to src/lib.rs: pub mod private_room_tracker; (behind #[cfg(feature = "net")])

3. Write tests:
   - PrivateRoomTracker can publish and discover via InMemoryDiscoveryBackend
   - Two private-room trackers on the same discovery secret discover each other
   - Two private-room trackers on different discovery secrets do NOT discover
     each other (namespace isolation)
   - Self-filter works (own endpoint excluded)

4. The test helpers follow the pattern in public_room_tracker.rs tests and
   test_public_lobby_integration.rs.

Deliverables:
- src/private_room_tracker.rs (feature-gated behind "net")
- Public API: new(), publish_once(), discover_once(), shutdown(),
  identity(), local_endpoint_id()
- Unit tests

Acceptance criteria:
- PrivateRoomTracker wraps PublicRoomTracker with zero code duplication
  for publish/discover logic.
- Two peers on the same discovery secret discover each other.
- Two peers on different discovery secrets are isolated.
- All tests pass without touching the real DHT.
- cargo build --features net succeeds.

CARD 7: ROOM JOIN — DHT DISCOVERY ON TICKET JOIN
================================================================================

Title: Perform DHT discovery when joining from a DHT-enabled ticket
Status: TODO
Priority: HIGH
Dependencies: Card 2 (Ticket Extension), Card 5 (Private-room Tracker)

Objective:
When parsing a ticket that contains a discovery_secret, perform a DHT
lookup before subscribing to the gossip mesh.  Merge discovered peer
addresses with the ticket's explicit peers for bootstrap.

Tasks:

Part A — Ticket join in chat_core / frontends:

1. In examples/chat.rs — the ticket join flow (~line 700):
   - After parsing the ticket, check if discovery_secret is Some.
   - If yes, perform a DHT lookup using PrivateRoomTracker.
   - Merge the discovered peers (EndpointIds) with the ticket's explicit
     peers and RoomStore peers.
   - Seed MemoryLookup with both the ticket's explicit addresses and
     (if possible) resolved addresses for DHT-discovered peers.

   Key: DHT only returns EndpointIds (32-byte public keys), not full
   EndpointAddr(including relay/direct addresses).  The discovered peers
   need address resolution.  This is where GossipAddressLookup and
   iroh's address resolution come in.

   When we call gossip.subscribe(topic, discovered_ids), iroh uses its
   address lookup chain (MemoryLookup -> GossipAddressLookup -> mDNS ->
   DHT/Pkarr) to resolve the public keys into addresses.  So simply
   adding the EndpointIds to the bootstrap list and subscribing should
   work — iroh's address resolution finds them.

   But to seed MemoryLookup, we need addresses.  For DHT-discovered peers
   we don't have addresses yet.  That's fine — iroh's address lookup
   handles that.  We only seed MemoryLookup from ticket addresses.

2. In examples/iced_chat/app.rs — the ticket join handler:
   - Check discovery_secret on the parsed ticket.
   - If present, call discover_once() and merge discovered peer IDs
     into the bootstrap list before subscribing.

Part B — Error handling:

1. If DHT lookup fails (timeout, no net), fall back to ticket peers only
   (current behaviour).  Log a warning.

2. If DHT lookup returns peers, log success.  If some peers were already
   in the ticket's peer list, avoid duplicates (collect_bootstrap_peers
   already handles this).

Deliverables:
- Modified ticket join paths:
  - examples/chat.rs join flow
  - examples/iced_chat/app.rs ticket join handler
- Graceful fallback when DHT is unavailable

Acceptance criteria:
- Joining a DHT-enabled ticket performs a DHT lookup.
- Discovered peers are merged with ticket peers for bootstrap.
- Joining a legacy ticket (no discovery_secret) works exactly as before.
- When DHT lookup fails, joining falls back to ticket peers.
- cargo build --features examples succeeds.

   - Lifecycle: create -> publish_once -> discover_once -> shutdown.
   - Integration: used with ContinuousTracker for background loops.

   - Existing rooms work unchanged (no discovery_secret in room.json).
   - New rooms with DHT get a discovery_secret.
   - No manual migration needed — the code handles both.

5. Architecture documentation:
   - Create or update docs/ARCHITECTURE.md (or ARCHITECTURE.md at repo
     root) with:
     - How discovery works: ticket carries discovery_secret -> DHT
       lookup -> discovered peers merged into gossip bootstrap.
     - Component diagram: user -> frontend -> PrivateRoomTracker ->
       TopicDiscoveryBackend -> distributed-topic-tracker -> Mainline DHT.
     - Flow diagram for room creation and join.
     - How ContinuousTracker maintains presence.

6. Security model:
   - Discovery secret secrecy: only shared via ticket (out-of-band or
     whisper).  Anyone with the secret can find the room's current peers.
   - No encryption: the record payload is the peer's public EndpointId
     (inherently public).
   - Topic privacy: the gossip topic is derived from a random value
     (the topic itself), so it cannot be guessed.
   - An attacker who knows the discovery_secret can discover current
     peers but cannot:
     a) Decrypt room messages (unless they also have the topic key)
     b) Impersonate a peer (need the peer's SecretKey)
     c) Read past records (minute-rotating keys)
   - The distributed-topic-tracker's HPKE encryption is an additional
     layer — the record payload is encrypted on the DHT.

7. Limitations:
   - DHT discovery only returns PeerIds, not full addresses.  iroh's
     address resolution (MemoryLookup -> GossipAddressLookup -> mDNS ->
     DHT/Pkarr) fills in the addresses after the peer ID is known.
   - DHT requires UDP access to the global Mainline DHT.  Firewalls,
     NAT, and corporate networks may block this.
   - Initial discovery may take up to 30 seconds (configurable) because
     the DHT lookup has a timeout.
   - The DHT discovery record is time-bound (current minute +/- 1),
     so a peer that publishes then immediately shuts down may not be
     discoverable after 2 minutes.
   - No offline mailbox integration for DHT-discovered peers (offline
     delivery is a separate feature).

8. Privacy model:
   - Room membership is discoverable by anyone with the ticket.
   - No central server tracks room membership.
   - DHT records are ephemeral (current minute +/- 1).
   - Room name/topic is not leaked on the DHT — only the discovery
     secret (a random 32-byte value) is used as the namespace.
   - The discovery protocol is not anonymous — it uses the peer's
     long-lived iroh EndpointId.

- Architecture documentation (docs/ARCHITECTURE.md or ARCHITECTURE.md)

================================================================================
REMAINING RISKS
================================================================================

1. DHT address resolution:
   The DHT only returns EndpointId (32-byte public key).  It does NOT
   return relay URLs or direct IP addresses.  iroh's address lookup
   chain must resolve these.  If iroh's built-in resolution fails
   (no relay, no mDNS, no Pkarr), the peer ID is known but the peer
   cannot be connected to.  This is a limitation of the DHT protocol,
   not something we can fix in boru-chat.

2. DHT port/firewall:
   The Mainline DHT uses UDP on a random ephemeral port.  Corporate
   firewalls, NAT configurations, and containerised environments may
   block DHT traffic.  The discovery feature degrades gracefully
   (falls back to ticket-only), but the user loses the benefit.

3. DHT rate limits:
   Frequent publishing (sub-minute) on the Mainline DHT is considered
   antisocial.  The defaults (publish every 5 minutes) are conservative,
   but users who override them may be rate-limited by the DHT network.

4. Sequence number semantics:
   The distributed-topic-tracker uses i64::MAX as the sequence number
   for all records (see record.rs line 328).  This means the last
   published record always wins, regardless of actual publication order.
   For our use case (publishing EndpointId), this is fine — stale
   records expire anyway.

5. Room secret size:
   The DiscoverySecret is 32 bytes.  When embedded in a base32 ticket,
   it adds ~52 characters to the ticket string.  Existing tickets are
   ~85 characters.  DHT-enabled tickets become ~137 characters.
   This may affect copy/paste workflows on mobile or character-limited
   channels.

6. Concurrent DHT instances:
   Each room with DHT discovery creates a new PrivateRoomTracker (and
   potentially a new DHT client).  Multiple DHT clients on the same
   machine share the global Mainline DHT but each maintains its own
   routing table.  This is wasteful.  Future optimisation: share a
   single DHT client across all room trackers.

7. Topic stability:
   Currently, room TopicIds are random.  If a user recreates a room
   (deletes room.json), they get a different topic.  With DHT discovery,
   the topic is still random.  The discovery secret is what enables
   DHT lookup, not the topic.  This means a room's ticket changes if
   the creator recreates the room — but that's already the case even
   without DHT.

8. ContinuousTracker lifetime:
   The ContinuousTracker holds a CancellationToken and background tasks.
   If the IcedChat state is dropped without calling shutdown(), the
   tasks are aborted (JoinHandle is not detached).  Current code already
   handles this for the public lobby tracker via the Option<ContinuousTracker>
   field.  The per-room HashMap<..., ContinuousTracker> must also clean
   up on drop.

FUTURE WORK (PUBLIC ROOMS — NOT IN SCOPE)
================================================================================

This roadmap covers PRIVATE-ROOM DHT discovery only.  Public-room
discovery already exists in the codebase (see public_room.rs,
public_room_tracker.rs, public_room_continuous.rs, etc.) and operates
on a different discovery key derivation (deterministic from room name).

Public-room enhancements that are explicitly NOT in scope:

1. Public room list / directory — no way to enumerate all public rooms.
2. Public room search — no name-based DHT lookup (discovery key is
   deterministic, so you need to know the room name).
3. Public room access control — all peers with the discovery key can
   find each other.
4. Public room moderation — no operator/administrator concept.
5. Public room persistent history — no DHT-backed message archive.

These are separate features that build on the existing public-room
discovery foundation.

For the purpose of this roadmap, all "discovery" refers to PRIVATE-ROOM
discovery using a random discovery secret carried in the ticket.
