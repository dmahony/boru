You are acting as a senior Rust reviewer performing a comprehensive
implementation audit.

Your task is not to add new features.

Your task is to verify whether the MCP Diagnostics implementation
matches its design specification and to identify anything that is
incomplete, incorrect, fragile, insecure, or inconsistent.

Assume the implementation is already present in the repository.

Your goal is to produce a detailed audit and, if necessary, an
implementation plan for missing work.

------------------------------------------------------------------------

RULES

Do not immediately start modifying code.

First inspect the entire implementation.

Understand the architecture before making any judgement.

If something appears to be missing, confirm that it truly does not exist
before reporting it.

Do not assume a feature is absent simply because it is implemented
differently.

Accept equivalent implementations if they satisfy the design goals.

------------------------------------------------------------------------

THINGS TO INSPECT

Inspect:

-   diagnostics module
-   MCP server
-   room discovery
-   peer tracking
-   room membership
-   message send path
-   message receive path
-   probe implementation
-   event recording
-   serialization
-   tests
-   documentation
-   CLI integration
-   configuration
-   security

Understand how all of these interact.

------------------------------------------------------------------------

ARCHITECTURE REVIEW

Verify that:

✓ diagnostics is reusable ✓ diagnostics is not tightly coupled to Iced
widgets ✓ MCP is only an adapter over reusable diagnostics code ✓ GUI
and MCP share the same underlying application logic ✓ diagnostics
failures cannot break normal chat ✓ diagnostics is thread-safe ✓
diagnostics uses bounded memory ✓ diagnostics does not duplicate
existing application state unnecessarily

------------------------------------------------------------------------

DIAGNOSTICS SERVICE

Verify:

-   bounded event storage
-   bounded received probe storage
-   monotonic event sequence numbers
-   timestamp generation
-   serialization
-   event filtering
-   room filtering
-   sequence filtering
-   eviction of oldest events
-   thread safety
-   cloning
-   deterministic behaviour

Look for race conditions. Look for deadlocks. Look for unnecessary
locking.

------------------------------------------------------------------------

DISCOVERY INSTRUMENTATION

Verify that instrumentation exists for every implemented stage.

Determine which stages are observable. Determine which stages are
inferred. Determine which stages are guessed.

The implementation should never invent diagnostic information.

If a stage cannot be observed it should report: - Unknown - NotObserved

rather than guessing.

------------------------------------------------------------------------

PEER STATE

Verify peer state contains useful information.

Check: - discovery sources - addresses - connection state - room
membership - timestamps - last errors

Ensure peer state updates deterministically.

------------------------------------------------------------------------

PROBE IMPLEMENTATION

Verify probes: - use the normal message path - use existing
serialization - use existing signing - preserve probe IDs - preserve
hashes - are not displayed as normal chat - cannot execute commands -
cannot corrupt chat state

Verify duplicate handling. Verify latency calculation. Verify clock skew
handling.

------------------------------------------------------------------------

MCP TOOLS

Verify every implemented MCP tool.

Check: - schema - validation - error handling - timeout behaviour -
limits - returned JSON - deterministic output

Ensure MCP never exposes: - secret keys - mailbox keys - private room
tickets - filesystem - arbitrary command execution

------------------------------------------------------------------------

AUTONOMOUS DIAGNOSTICS

Verify the AI can determine:

peer discovered ↓ address resolved ↓ connection established ↓ room
joined ↓ topic membership ↓ probe broadcast ↓ probe received

without guessing.

If any stage cannot currently be determined, identify exactly why.

------------------------------------------------------------------------

FAILURE CLASSIFICATION

Inspect the classifier.

Verify it produces deterministic results.

Look for: - incorrect priorities - incorrect assumptions - false
positives - false negatives - ambiguous cases - missing edge cases

Ensure “Unknown” is used whenever evidence is insufficient.

------------------------------------------------------------------------

WAITING IMPLEMENTATION

Inspect any waiting APIs.

Verify they: - don’t busy poll - wake efficiently - handle
cancellation - handle timeout - handle lagging subscribers

------------------------------------------------------------------------

SECURITY REVIEW

Verify: - loopback default binding - safe defaults - no secret leakage -
no arbitrary filesystem - no unrestricted command execution - no unsafe
deserialization - no obvious denial-of-service risks - bounded
allocations - bounded queues - bounded event history

------------------------------------------------------------------------

PERFORMANCE REVIEW

Identify: - unnecessary cloning - large allocations - repeated
serialization - expensive locking - contention - unnecessary copies -
memory growth - hot paths

Recommend improvements where appropriate.

------------------------------------------------------------------------

CODE QUALITY

Review: - naming - module layout - documentation - comments -
consistency - error handling - logging - tracing - testability - public
API quality

------------------------------------------------------------------------

TESTS

Verify tests exist for: - event ordering - bounded buffers - probe
serialization - duplicate probes - latency - timeouts - room filtering -
peer discovery - classification - probe delivery - MCP responses -
security

Look for missing edge cases.

Suggest additional tests if useful.

------------------------------------------------------------------------

DOCUMENTATION

Verify documentation covers: - architecture - security - setup - MCP -
probe workflow - two-node testing - limitations - future work

------------------------------------------------------------------------

PRACTICAL REVIEW

Determine whether an AI agent could realistically diagnose: - peer
discovery failures - asymmetric discovery - room join failures - topic
membership failures - probe delivery failures - asymmetric delivery

using only the implemented MCP interface.

If not, explain exactly what information is still missing.
