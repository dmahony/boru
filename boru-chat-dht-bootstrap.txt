Boru Chat Automatic DHT Bootstrap — Task Breakdown

Repository:

https://github.com/dmahony/iroh-gossip-chat

Overall goal:

Replace endpoint-bearing room tickets with stable room invitations. A room invitation should contain only the room topic, a random discovery secret and a version identifier. Active peers should discover one another through distributed-topic-tracker, while boru-chat continues using its existing gossip implementation.

Do not replace boru-chat gossip with upstream iroh-gossip.

Task 1: Audit the existing bootstrap flow

Inspect the repository and document the current room bootstrap lifecycle.

Identify:

where Ticket is defined
how tickets are encoded and decoded
where room topics are generated
how RoomStore persists topics and peers
how MemoryLookup is populated
where subscribe and subscribe_and_join are called
how bootstrap peers are refreshed
how the TUI and iced GUI create and join rooms
which tests cover stale bootstrap peers and multi-peer joins

Also inspect the current public API and source of distributed-topic-tracker.

Confirm:

whether it can be used with default features disabled
which DHT, record, crypto and publisher types are publicly exposed
whether its record publishing and lookup logic is coupled to upstream iroh-gossip
supported versions of iroh, Tokio, mainline and ed25519-dalek
whether dependency version conflicts will occur
crate licensing

Deliver a short architecture note before changing code.

Acceptance criteria:

existing bootstrap flow is clearly documented
integration points are identified
dependency compatibility is confirmed
any required patch or fork is justified before implementation
Task 2: Add a versioned stable room invitation

Create a new invitation type similar to:

pub struct RoomInviteV2 {
    pub version: u8,
    pub topic: boru_chat::proto::TopicId,
    pub discovery_secret: [u8; 32],
}

Requirements:

encode as a compact copy-paste-safe string
use a prefix such as boru1:
include no endpoint addresses
include no relay URLs
include no creator EndpointId
validate version and exact payload length
return clear errors for malformed input
redact the discovery secret from Debug
do not log full invitation contents at normal log levels

Keep the legacy ticket type unchanged for now.

Tests:

invitation round trip
invalid prefix
unsupported version
truncated payload
corrupted payload
exact payload length enforcement
debug output does not expose the secret

Acceptance criteria:

valid invitations encode and decode reliably
malformed invitations are rejected
invitation contents contain no endpoint information
Task 3: Extend room persistence

Extend RoomStore to persist:

room topic
random 32-byte discovery secret
optional cached last-known peers

Add migration for existing room files that do not contain a discovery secret.

Migration behaviour:

preserve the existing topic
preserve cached peers
generate the discovery secret once
save the migrated room atomically
reuse the same secret on all future launches
never silently replace an existing secret

Use secure randomness.

Tests:

new room gets a discovery secret
discovery secret survives restart
legacy room store migrates successfully
topic remains unchanged after migration
cached peers remain unchanged
migration does not regenerate the secret twice
malformed room files fail safely

Acceptance criteria:

existing users do not lose their room
each room has a stable discovery secret
room migration is backward compatible
Task 4: Add deterministic tracker namespace derivation

Implement a function that derives the DHT tracker namespace from raw room topic bytes.

Use domain separation conceptually equivalent to:

SHA-256("boru-chat room discovery v1" || topic_bytes)

Requirements:

use raw topic bytes rather than display strings
keep the domain separator fixed and documented
return the type expected by distributed-topic-tracker
do not derive the discovery secret from the room topic
do not derive private room secrets from room names

Tests:

same topic produces same namespace
different topics produce different namespaces
changing the domain separator changes the result
output is stable across restarts and platforms

Acceptance criteria:

namespace derivation is deterministic
domain separation is explicit
no string-format dependency exists
Task 5: Add the dependency and build a minimal tracker wrapper

Add distributed-topic-tracker using the smallest compatible feature set.

Prefer:

distributed-topic-tracker = {
    version = "...",
    default-features = false
}

Create a discovery module such as:

src/discovery/
├── mod.rs
├── invite.rs
└── topic_tracker.rs

Implement a minimal wrapper that can:

initialise the tracker DHT
publish the local EndpointId
query records for the room
decrypt and validate records
return discovered EndpointIds
shut down cleanly

Do not integrate it into application startup yet.

Suggested API:

pub struct TopicTracker {
    // lifecycle and cancellation state
}

impl TopicTracker {
    pub async fn start(
        endpoint: iroh::Endpoint,
        topic: boru_chat::proto::TopicId,
        discovery_secret: [u8; 32],
    ) -> Result<Self>;

    pub async fn publish_once(&self) -> Result<()>;

    pub async fn discover_once(&self) -> Result<Vec<iroh::EndpointId>>;

    pub async fn shutdown(self) -> Result<()>;
}

Acceptance criteria:

the project compiles with the dependency
boru-chat still uses its existing gossip crate
local EndpointId records can be published and read
the wrapper shuts down without leaked tasks
Task 6: Implement discovery record validation

Ensure discovered records are validated before peer IDs are returned.

Reject:

malformed records
records that fail decryption
invalid signatures
stale records
future records outside an allowed clock tolerance
records whose signing identity does not match the advertised EndpointId
oversized records
duplicate peers
the local EndpointId
records exceeding configured peer limits

Use the tracker crate’s native record format and validation wherever possible.

Do not invent a second incompatible record protocol unless the dependency cannot support the required properties.

Tests:

valid record accepted
invalid signature rejected
wrong room secret rejected
stale record rejected
future record rejected
EndpointId/signing identity mismatch rejected
self-record filtered
duplicate records deduplicated
excessive peer entries rejected or bounded

Acceptance criteria:

untrusted DHT data cannot flow directly into gossip joins
record processing is bounded
private room records require the discovery secret
Task 7: Create an injectable discovery backend for tests

Introduce an abstraction around DHT publishing and lookup so tests do not depend on the public Mainline DHT.

For example:

#[async_trait]
pub trait TopicDiscoveryBackend {
    async fn publish(&self, record: DiscoveryRecord) -> Result<()>;

    async fn lookup(
        &self,
        namespace: TrackerNamespace,
    ) -> Result<Vec<EncryptedDiscoveryRecord>>;
}

Provide:

production Mainline DHT implementation
in-memory or mocked test implementation

The production implementation should wrap distributed-topic-tracker.

Acceptance criteria:

tracker logic can be tested without internet access
CI does not depend on the live BitTorrent DHT
production and test backends use the same validation path
Task 8: Integrate initial discovery into room startup

Update startup to:

load or create the room topic and discovery secret
build the Iroh endpoint
add existing address lookup providers
start the router and boru gossip protocol
start the topic tracker
perform an initial discovery lookup
merge discovered EndpointIds with cached or legacy peers
populate MemoryLookup with any stored full endpoint information
subscribe to the room topic
join discovered peers when available
remain subscribed when no peers are found
publish the local discovery record

Important behaviour:

no DHT results must not be fatal
DHT failure must not prevent room startup
the room creator should subscribe and wait for inbound peers
remove the fatal behaviour around a fixed 30-second initial join timeout
stale cached peers should not terminate the process
emit clear degraded-mode status messages

Acceptance criteria:

a room starts successfully with zero discovered peers
a peer can bootstrap from discovered EndpointIds
cached peers continue to work
DHT failure produces a warning, not application exit

impl TopicHandle {
    pub async fn join_peers(
        &self,
        peers: impl IntoIterator<Item = EndpointId>,

Requirements:

deduplicate current neighbours
ignore the local EndpointId

bound concurrent joins

emit tracing events for attempts and results

Tests:

duplicate peer is not joined twice

concurrency limit is respected

Acceptance criteria:

tracker results can be applied after startup
joining new peers does not require recreating the subscription

Task 13: Preserve legacy ticket compatibility

Implement format detection:

boru1:...  -> new stable room invitation
other supported format -> legacy ticket

When joining with a legacy ticket:

extract the topic
use supplied endpoint information as initial cached bootstrap data
generate a discovery secret if the legacy format has none
clearly document that a newly generated secret cannot be known by existing peers unless it is shared

Choose and document one migration strategy:

legacy tickets remain legacy-only until the room owner shares a new invitation; or
connected peers negotiate and distribute the room discovery secret securely.

Do not silently assume independently generated secrets will match.

Acceptance criteria:

legacy tickets still decode
migration behaviour is explicit
no incompatible discovery secret is silently created and treated as shared
Task 14: Resolve legacy room-secret migration between peers

Design and implement a secure way for an existing legacy room to adopt one shared discovery secret.

Preferred options:

room owner explicitly upgrades and shares a new stable invite; or
an authenticated gossip/control message distributes the secret to already-authorised peers

Do not broadcast the discovery secret over an unauthenticated public topic.

Document:

who is allowed to initiate migration
how conflicting secrets are handled
how peers persist the accepted secret
whether migration can be revoked
how old clients behave

Acceptance criteria:

all upgraded peers use the same discovery secret
migration cannot be triggered by an unauthenticated outsider
conflicting migrations are handled deterministically
