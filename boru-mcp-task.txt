You are responsible for troubleshooting the running Boru Chat GUI application using its connected MCP diagnostic and GUI-testing tools, then creating actionable follow-up tasks on the user’s existing Kanban board.
use the boru-mcp profile for this
Do not modify source code during this workflow unless the user explicitly asks you to implement a fix.

Your responsibilities are:

1. Test the live application.
2. Gather structured evidence from both LAN clients.
3. Identify the first failed stage.
4. Inspect source code only when necessary to explain the observed failure.
5. Create well-scoped follow-up Kanban tasks for confirmed issues.
6. Avoid duplicate Kanban tasks.

# Available systems

You may have access to:

* Boru MCP server for GUI client A on host lubuntuVM-001
* Boru MCP server for GUI client B on host lubuntuVM-002
* Boru networking diagnostics
* Boru GUI diagnostics
* Boru GUI action tools
* Boru probe and message-test tools
* repository source code
* an existing Kanban board through connected tools

Use the live MCP data as the primary source of truth for runtime behaviour.

# Safety and operating rules

* Do not guess.
* Do not report a stage as failed merely because it was not observable.
* Use `unknown` or `not_observed` when evidence is insufficient.
* Do not expose secret keys, private tickets, discovery secrets, mailbox keys, message contents, or arbitrary files.
* Do not run destructive GUI actions.
* Do not delete, close, archive, or modify existing Kanban tasks unless explicitly required.
* Do not create duplicate tasks for issues already present on the board.
* Do not create tasks for transient warnings unless they represent a reproducible defect or a clear observability gap.
* Prefer a small number of well-scoped tasks over many vague tasks.

# Phase 2: Establish the test topology

Query node status on both clients.

Record:

* client label
* node ID
* application version
* MCP availability
* GUI-test actions enabled or disabled
* active rooms
* latest diagnostic sequence
* current GUI screen

Query room and GUI state on both clients.

Identify a room that is active on both clients.

If there is no shared room:

1. Use approved semantic GUI actions to open or join the intended existing room when possible.
2. Do not use arbitrary tickets or create unrelated rooms.
3. If a shared room still cannot be established, stop the delivery test and classify the failure as `local_room_unavailable`.

# Phase 3: Run discovery diagnostics

Test both directions independently:

```text
Client A observing Client B
Client B observing Client A
```

For each direction, determine:

* expected peer discovered
* discovery source
* addresses observed
* address resolution status
* connection attempted
* connection established
* subscription joined
* peer visible as a topic member
* last successful stage
* first failed stage
* relevant errors

Use the autonomous discovery-test tool where available.

Retrieve the relevant diagnostic event ranges from both clients.

Check for asymmetric behaviour, including:

* A discovers B but B does not discover A
* one client resolves addresses while the other does not
* one client connects while the other remains disconnected
* one client reports topic membership while the other does not
* peers repeatedly join and leave
* discovery uses relay when a direct LAN path was expected
* a peer is present in network state but missing from GUI state

# Phase 4: Run probe delivery tests

Send a unique diagnostic probe from A to B.

Record:

* probe ID
* sender node ID
* room ID
* message hash
* send timestamp
* broadcast result

On B, confirm:

* exact probe ID received
* matching message hash
* sender identity
* receive timestamp
* duplicate count
* latency when valid
* network receipt
* Iced application handling
* application-state update
* GUI diagnostic-state update
* whether the probe was intentionally hidden

Repeat the complete test from B to A.

Do not treat an intentionally hidden diagnostic probe as a GUI failure.

# Phase 5: Run a normal GUI message workflow

When GUI actions are enabled, test the same path a person uses.

For each direction:

1. Open the shared room through the semantic GUI action.
2. Wait for the expected room-selection state.
3. Set composer text to a unique test message.
4. Wait for the composer state to update.
5. Submit through the normal GUI action.
6. Verify that the GUI action completed.
7. Verify that the composer cleared when expected.
8. Verify that the local application created the message.
9. Verify that the local visible GUI state reflects the message.
10. Verify that the remote client received the exact test message or correlated message hash.
11. Verify that the remote application state updated.
12. Verify remote GUI visibility when the room is selected and the message should be visible.

Use a unique identifier such as:

```text
BORU-GUI-TEST-<timestamp-or-generated-id>
```

Do not reuse an old test identifier.

# Phase 6: Run consistency checks

On both clients, compare:

```text
Network state
Application state
GUI diagnostic state
Visible GUI state
```

Look for confirmed inconsistencies such as:

* room exists in application state but is absent from the GUI room list
* peer exists in application state but is absent from the visible peer list
* message exists in application state but is absent from the selected conversation
* network event arrived but Iced did not handle it
* Iced handled the event but application state did not update
* application state updated but the GUI snapshot remained stale
* the selected room or conversation refers to missing state
* counts disagree without an intentional filter explaining the difference
* GUI action was queued but never handled
* GUI action was handled but expected state never appeared

# Phase 7: Classify each issue

For every reproducible issue, identify the first failed stage from this ordered list:

```text

probe_broadcast
probe_delivery
gui_action_queue
gui_action_handling
network_to_iced_event
application_state_update
gui_state_update
message_visibility
unknown
```

For each issue, provide:

* concise title
* affected direction: A→B, B→A, or both
* reproducibility
* expected behaviour
* actual behaviour
* first failed stage
* furthest successful stage
* supporting MCP evidence
* relevant event sequence range
* likely source modules, when source inspection supports this
* confidence: high, medium, or low

Do not convert speculation into a confirmed defect.

# Phase 8: Inspect the existing Kanban board

4. Do not create a duplicate when an equivalent actionable task already exists.

* acceptance-criteria format

# Phase 9: Create follow-up Kanban tasks

Create a task only for:

* a confirmed reproducible defect
* a partially implemented diagnostic stage that materially prevents troubleshooting
* a missing automated test that would have caught a confirmed issue
* a clear security or stability problem found during the workflow
* a concrete documentation gap that blocks repeatable testing

Do not create tasks for:

* successful tests
* one-off connection delays that recover normally
* speculative redesigns
* vague “investigate networking” work
* issues already represented on the board

Create separate tasks when fixes are independently actionable.

Combine closely related symptoms when they have the same root cause.

# Kanban task format

Use a concise title such as:

```text
Fix mDNS peer discovery not reaching topic membership
```

The task description should contain:

## Problem

A concise description of the confirmed defect.

## Reproduction

Numbered steps using the actual GUI and MCP workflow.

## Expected

The expected application behaviour.

## Actual

The observed behaviour.

## Diagnostic evidence

Include:

* affected nodes
* affected direction
* room ID in safe abbreviated form when useful
* peer IDs in safe abbreviated form
* failed stage
* furthest successful stage
* relevant event types and sequence numbers
* error codes
* probe ID or message hash in abbreviated form
* whether the issue is reproducible

Do not include secrets, full tickets, or message contents.

## Likely area

List likely files or modules only when supported by source inspection.

## Acceptance criteria

Provide testable completion criteria.

Example:

* Both LAN clients discover each other through the expected discovery mechanism.
* Both clients reach topic membership within the supported timeout.
* A→B and B→A diagnostic probes succeed.
* The relevant automated regression test passes.
* Existing discovery and message tests remain passing.

## Suggested tests

List specific unit, integration, or two-client tests that should be added or updated.

## Priority

Set priority using evidence:

* Critical: security issue, data loss, or application unusable
* High: core discovery or bidirectional messaging consistently broken
* Medium: partial failure, GUI inconsistency, weak error handling, or missing observability
* Low: documentation or non-blocking cleanup

Do not mark everything high priority.

# Task decomposition rules

Create the smallest task that can be completed and verified independently.

Examples of good separation:

* Fix mDNS result not being inserted into the Iroh address lookup
* Fix peer membership event not updating the Iced peer list
* Add regression test for asymmetric A→B probe delivery
* Expose connection failure reason in MCP peer diagnostics

Examples of tasks that are too broad:

* Fix networking
* Improve MCP
* Make discovery reliable
* Test everything
* Refactor the GUI

# Final report

After troubleshooting and Kanban updates, return:

## Test summary

* clients tested
* shared room
* directions tested
* discovery result
* topic-membership result
* probe result
* normal GUI-message result
* GUI consistency result

## Confirmed issues

For each issue:

* title
* failed stage
* affected direction
* confidence
* brief evidence

## Kanban actions

List:

* tasks created
* matching existing tasks found
* tasks skipped as duplicates
* issues not converted into tasks and why

## Overall result

One of:

```text
pass
partial_pass
fail
inconclusive
```

Use:

* `pass` only when discovery, topic membership, probes, and normal GUI messaging work in both directions
* `partial_pass` when core connectivity works but one direction or one GUI stage fails
* `fail` when discovery, joining, or delivery is reproducibly broken
* `inconclusive` when required MCP evidence or one client is unavailable
